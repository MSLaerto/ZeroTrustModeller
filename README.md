# ZeroTrustModeller

**ZeroTrustModeller** — программа для автоматизированного анализа IoT-инфраструктуры и генерации планов внедрения модели нулевого доверия (Zero Trust). Разработана с учётом требований российского законодательства (ФСТЭК, приказы №239 и №31, ФЗ-152, ФЗ-187) и специфики местного рынка (импортозависимость, кадровый дефицит).

## Возможности

- **Инвентаризация IoT-устройств** с ручным и автоматизированным режимом (через Nmap).
- **Автоматический сбор данных из сети** (`nmap` + парсер) → заполнение базового JSON за минуты.
- **Доопределение бизнес-контекста**: критичность, категория КИИ, использование отечественных алгоритмов, логические группы взаимодействия.
- **Расчёт индекса готовности к Zero Trust (RZT)** — от 0 до 100%, учитывающий шифрование, сегментацию, MFA, долю импортных устройств.
- **Генерация пошагового плана внедрения** для малого бизнеса и корпораций.
- **Поддержка отраслевых кейсов**: умный дом, офис, больница, объект ЖКХ, логистический центр.
- **Интеграция с российской нормативной базой** (ГОСТ Р 34.12-2015 «Кузнечик», Рутокен, КриптоПро, Яндекс.Ключ).
- **Встроенная таксономия кибератак на IoT** для риск-ориентированного планирования.

## Как пользоваться программой (пошагово)
1. Получите исходные данные об устройствах (два варианта)
### Вариант А (автоматическое сканирование) — рекомендуется для новых проектов или быстрой инвентаризации.

2. В главном меню выберите пункт 13 «Сканировать сеть через Nmap и импортировать устройства».

3. Укажите диапазон IP (например, 192.168.1.0/24).

4. Программа запустит nmap, распарсит результат и создаст базовый файл data/infrastructure.json.

5. Что вы получите после сканирования:

 - IP-адреса, открытые порты, протоколы.

 - Автоматическое определение типа устройства (камера, роутер, датчик, принтер и т.п.).

 - Заполненные поля: type, category, ports, protocols, encryption (факт поддержки/использования TLS/HTTPS/SSH).

**Важно**: автоматика не умеет определять:

 - критичность устройства (critical);

 - категорию КИИ (kii_category);

 - использование отечественных алгоритмов (uses_domestic_algorithm);

 - возможность смены протоколов (can_change_protocols);

 - вычислительную мощность, память, объём данных (compute_power, memory, data_volume);

 - логическую группу взаимодействия (interaction_group) — по умолчанию ставится "default".

**Поэтому следующий шаг обязателен:**

### Вариант Б (ручное создание JSON) — для проектов, где известен полный состав устройств, или для внесения правок после сканирования.

1.1 спользуйте пункт меню 1 «Инвентаризация устройств (Этап 1)».

#### Вы можете добавить устройство из каталога (уже содержит типовые характеристики) или создать пользовательское.

Подробно заполните все поля, особенно:

- критичность,

- группа взаимодействия (см. ниже),

- категория КИИ,

- технические параметры (вычислительная мощность, память, объём данных).

2. Доопределите бизнес-атрибуты устройств (критично для точности RZT)
После первичного импорта (автоматического или ручного) обязательно зайдите в инвентаризацию и уточните:


- critical	true – если выход устройства из строя остановит критический бизнес-процесс.
- kii_category	Укажите согласно ФЗ-187: 1–2 для объектов КИИ, 0 – не относится.
- uses_domestic_algorithm	true – если используется ГОСТ Р 34.10-2012, «Кузнечик» или сертифицированное СКЗИ.
- can_change_protocols	«да» – если устройство позволяет переключить HTTP → HTTPS, Telnet → SSH.
- compute_power, memory, data_volume	Оцените как «низкая», «средняя» или «высокая» (влияет на выбор алгоритмов шифрования).
- interaction_group	Важный шаг! Объедините устройства, которые должны общаться друг с другом. Например: outdoor_cameras, climate_sensors, control_room. Это основа для последующей сегментации сети.

####  **Подсказка**: правильная группировка взаимодействия — основа для сегментации. Если все устройства в группе "default", прототип покажет критический риск (отсутствие сегментации).

3. Запустите анализ и получите план внедрения
После того как инфраструктура описана качественно (автоматика + ручные правки), выполните последовательно:

- Анализ шифрования (Этап 2) – выявит устройства без шифрования и предложит отечественные криптоалгоритмы.

- Планирование сегментации (Этап 3) – предложит изолированные VLAN на основе групп взаимодействия.

- Анализ MFA (Этап 4) – определит, где нужна многофакторная аутентификация (особенно для КИИ и критических устройств).

- Планирование мониторинга (Этап 5) – рекомендации по SIEM, поведенческому анализу.

- После этого выберите пункт «6. Анализ индекса Zero Trust (RZT)» – вы увидите текущий уровень защищённости (0–100%) и прогноз после внедрения рекомендованных мер.

4. Сформируйте отчёт
Пункты 7–11 – отчёты по отдельным этапам.

Пункт 12 – полный отчёт (инвентаризация + шифрование + сегментация + MFA + мониторинг + RZT).

Отчёты сохраняются в папке reports/ в виде текстовых файлов.

## Результаты (индекс RZT)
Индекс RZT  рассчитывается по формуле:

 ``` RZT = 0.3·SE + 0.3·MFA + 0.2·E + 0.2·(100% - IM) ```
где:

SE – уровень сегментации (доля устройств в корректных VLAN/группах);

MFA – доля устройств с включённой многофакторной аутентификацией;

E – доля устройств с шифрованием трафика (TLS, SSH, MQTTS);

IM – доля устройств импортного производства.

Интерпретация:

≥80% – продвинутый уровень, полное соответствие Zero Trust.

70–79% – высокий уровень, безопасность достаточна для большинства организаций.

60–69% – средний уровень, требуются улучшения.

50–59% – минимально удовлетворительно, настоятельно рекомендуется принять меры.

<50% – недостаточный уровень, срочное внедрение критически важно (особенно для КИИ).

### Советы для получения максимально точных результатов
Начните с автоматического сканирования (пункт 13) – это сэкономит часы ручного ввода.

Обязательно вручную проставьте critical и kii_category – автоматика этого не знает.

Не пренебрегайте группой взаимодействия (interaction_group) – от неё зависит качество рекомендаций по сегментации.

Для устройств без шифрования – прототип сам предложит алгоритм (ГОСТ «Кузнечик», «Магма») или укажет на необходимость замены.

Используйте каталог устройств (devices_catalog.json) – он уже содержит корректные значения compute_power, memory, data_volume, risk_level для многих популярных моделей (Hikvision, ОВЕН, MikroTik, Cisco и др.).

После каждого изменения инфраструктуры пересчитывайте индекс RZT (пункт 6) – он динамически обновляется.

Изучите примеры тестовых кейсов в папке test_data/ (умный дом, офис, больница, котельная, склад). Они демонстрируют «идеальное» заполнение JSON.
